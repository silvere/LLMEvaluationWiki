# 生产环境LLM评测

## 学术评测 vs 生产评测

学术 LLM 评测（MMLU、Chatbot Arena、SWE-bench 等）与生产环境中的 LLM 评测在目标、方法和约束上存在系统性差异。将学术排行榜分数直接用于生产决策是常见的错误。

| 维度 | 学术评测 | 生产评测 |
|-----|---------|---------|
| 评测目标 | 通用能力排名 | 特定业务任务质量 |
| 任务来源 | 标准化基准数据集 | 真实用户输入 |
| 成功标准 | 固定指标（准确率、胜率） | 业务指标（用户满意度、转化率） |
| 更新频率 | 模型发布时 | 持续/每次变更 |
| 评测人员 | 研究者 | 产品团队、用户 |

## 生产评测的核心挑战

**分布偏移**：
生产流量的分布与任何标准评测集都不完全匹配。用户的真实输入往往更杂乱、更模糊，包含领域专用词汇、错别字、多语言混合等。模型在标准基准上的高分不代表在真实用户输入上的高质量。

**真实答案缺失**：
生产环境中的大多数输入没有预定义的"正确答案"。开放式对话、创意生成、个性化建议等任务无法用传统准确率衡量，必须依赖代理指标（proxy metrics）或 LLM-as-Judge。

**延迟与质量权衡**：
生产评测必须同时考虑响应质量和响应速度。学术评测通常不限制延迟，但生产环境中用户体验对延迟高度敏感。

**版本管理复杂性**：
生产系统中的 LLM 应用涉及多个组件（提示词、模型、检索系统、后处理逻辑），任何一个组件的变更都可能影响整体质量，需要精细的版本追踪和 A/B 实验能力。

## 生产评测的实用框架

### 离线评测（Offline Evaluation）

在部署前，使用历史数据或构造的测试集进行评测：
1. **Golden Set**：人工精选的代表性测试用例集合，包含多种类型的输入和期望输出
2. **Regression Test**：已知模型表现良好的历史案例，用于检测回归
3. **Adversarial Set**：模型在早期版本中失败的困难案例

工具：DeepEval、promptfoo、LangSmith、Braintrust

### 影子评测（Shadow Evaluation）

将新版本模型的响应并行生成（但不展示给用户），与当前版本进行离线对比：
- 使用 LLM-as-Judge 对两个版本的响应进行成对比较
- 识别质量显著下降的用例子集
- 在完全部署前发现潜在问题

### 在线 A/B 实验

小比例流量（1%-10%）使用新版本模型，通过真实用户行为指标（点赞/踩、继续对话、任务完成率等）评测质量：
- 直接衡量业务影响，是最有效度的评测方式
- 需要足够的流量才能达到统计显著性
- 有暴露用户于潜在较差体验的风险

### 持续监控（Continuous Monitoring）

部署后持续监控关键质量指标：
- **自动告警**：当质量指标低于阈值时触发告警
- **人工抽样审核**：定期对随机样本进行人工评审
- **用户反馈捕获**：收集用户的显式（点赞/踩）和隐式（是否继续对话）反馈

工具：Arize Phoenix、LangSmith、Evidently AI、Weights & Biases

## 推荐的生产评测工具栈

对于大多数 LLM 应用团队，以下工具组合能覆盖主要评测需求：

- **离线评测**：DeepEval 或 promptfoo（CI/CD 集成）
- **实验追踪**：LangSmith 或 Braintrust（版本对比）
- **生产监控**：Arize Phoenix 或 LangSmith（追踪 + 告警）
- **人工评测**：LabelStudio 或 Scale AI 标注平台

## 关键原则

- **从真实用户数据构建评测集**：比任何人工设计的基准更贴近真实需求
- **评测驱动迭代**：每次提示词或模型变更前后都运行评测，而非临时性的一次性评测
- **定义业务指标**：找到与业务目标相关的代理指标，而非只追逐通用基准分数
