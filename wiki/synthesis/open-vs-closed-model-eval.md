# 开源与闭源模型评测的差异

## 核心差异维度

开源模型（如 LLaMA、Mistral、Qwen）和闭源模型（如 GPT-4、Claude、Gemini）在评测中面临根本性不同的条件，导致两者不能简单等同对待。

## 可访问性差异

**开源模型**：
- 权重公开，可在本地或私有环境部署
- 支持任意参数（温度、系统提示、批量大小）的精确控制
- 可访问模型内部状态（logits、attention 等），支持基于概率的评测指标
- 可进行无限次评测，不受速率限制和成本约束

**闭源模型**：
- 仅通过 API 访问，参数控制有限
- 无法访问 token 概率，不支持 perplexity 等基于概率的指标
- API 有速率限制和成本，大规模评测需要较高预算
- 模型可能随时更新，相同 API 名称在不同时间可能对应不同版本

## 可复现性问题

这是开源与闭源评测最关键的分歧点：

**开源模型评测**通常具有较高可复现性：固定模型权重 + 固定推理参数 → 相同的输出。评测框架（如 lm-evaluation-harness）可以精确记录所有参数，确保结果完全可复现。

**闭源模型评测**面临系统性挑战：
- **静默更新**：OpenAI、Anthropic 等公司不总是公告模型版本更新，同一模型名称的行为可能改变
- **温度随机性**：即使设置 temperature=0，部分模型的输出仍存在细微随机性
- **系统提示差异**：不同研究者使用的系统提示不同，导致结果不可比
- **数据截止日期**：训练数据截止日期不公开，数据污染程度难以评估

## 评测方式的不对称性

| 评测方式 | 开源模型 | 闭源模型 |
|---------|---------|---------|
| 概率/困惑度 | 支持 | 通常不支持 |
| 多选题（logit-based） | 支持 | 部分支持 |
| 生成式评测 | 支持 | 支持 |
| 微调后评测 | 可行 | 不可行 |
| 私有数据评测 | 可行（本地部署） | 存在数据隐私风险 |

## 排行榜设计对两者的影响

Open LLM Leaderboard 明确**不收录闭源模型**，原因在于：闭源模型的评测结果无法由社区独立验证，纳入后会降低整体可信度。这导致出现了两个平行的评测生态：
- 以 Open LLM Leaderboard 为核心的开源模型排名体系
- 以 Chatbot Arena 和 AlpacaEval 为核心的混合排名体系（同时包含开源和闭源模型）

## 评测成本不对称

- 开源 7B/13B 模型：在消费级 GPU 上即可完成全量评测，成本接近零
- 闭源 API 模型：完整评测一套主流基准的 API 成本可达数百至数千美元
- 这一成本不对称导致开源模型的评测数据远比闭源模型丰富

## 对研究者的建议

- 引用闭源模型评测结果时，明确记录模型版本（如 `gpt-4-0613` 而非 `gpt-4`）和评测日期
- 优先选择可自动复现的评测指标，减少对 API 端点的依赖
- 结合多个排行榜（包含不同评测方式）综合判断模型能力
- 对闭源模型的性能声明保持适度怀疑，尤其是在缺乏可独立验证信息时
