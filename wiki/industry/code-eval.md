# 代码与软件工程AI评测

> 代码AI评测从简单的函数生成延伸至真实仓库级别的软件工程任务，功能正确性与工程质量并重。

## 概述

代码生成是LLM最早展现实用价值的领域之一，也是评测体系发展最为成熟的垂直方向。早期评测以单函数代码生成为主，关注模型能否根据docstring生成功能正确的代码片段。随着模型能力提升，评测逐渐扩展到多文件项目、代码调试、测试用例生成，乃至完整的软件工程任务执行。

当前代码AI评测面临的核心张力是：通过测试用例不等于代码质量高，高质量的工程实践（可读性、安全性、性能、可维护性）难以被自动化指标捕捉。

## 主要评测维度

- **功能正确性**：生成的代码能否通过预设的单元测试，通常以pass@k指标衡量
- **代码质量**：可读性、注释规范、命名风格、复杂度等工程质量维度
- **安全漏洞**：生成代码是否包含SQL注入、缓冲区溢出等常见安全缺陷
- **多语言覆盖**：不同编程语言（Python、Java、C++、TypeScript等）下的表现差异
- **仓库级上下文理解**：在大型代码库中理解跨文件依赖、接口约定和架构风格的能力
- **测试用例生成**：能否为给定代码自动生成有效的边界测试和异常测试用例

## 代表性基准和数据集

- **HumanEval**：OpenAI发布的164道Python函数生成题，使用pass@k作为主要指标，是最广泛引用的代码基准
- **MBPP（Mostly Basic Python Problems）**：Google Research发布，974个入门级Python编程题，关注基础编程能力
- **APPS**：Hendrycks等人构建的竞赛级编程题数据集，包含10000道题，难度跨度从入门到竞赛
- **SWE-bench**：基于真实GitHub Issue和Pull Request的软件工程评测，要求模型在真实仓库中修复bug，是目前最接近真实工程场景的基准
- **CodeContests**：DeepMind构建的竞赛编程数据集，测试复杂算法推理能力
- **DS-1000**：数据科学代码生成基准，覆盖pandas、numpy、scikit-learn等主流库
- **CRUXEval**：代码执行推理评测，测试模型对代码运行结果的预测能力

## 评测挑战

**测试用例覆盖率问题**：HumanEval等基准的测试用例数量有限，模型可能通过恰好覆盖的少量测试但生成了功能不完整的代码，pass@k指标会高估真实能力。

**数据污染**：许多公开编程题已进入模型训练数据，静态基准的评测结果存在严重的数据泄露风险，SWE-bench等动态更新基准是应对方向之一。

**工程质量难量化**：安全性、可维护性等关键工程维度缺乏自动化评测手段，人工审查成本高且难以规模化。

**执行环境依赖**：仓库级代码评测需要配置复杂的运行环境，不同依赖版本可能导致同一代码在不同环境下测试结果不一致。

## 主要研究机构和团队

- **OpenAI**：HumanEval基准创建者，Codex模型推动了代码评测体系的标准化
- **Princeton NLP / Carlos E. Jimenez等**：SWE-bench的主要创建团队
- **DeepMind**：AlphaCode和CodeContests评测体系
- **BigCode Project**：开源代码模型和配套评测基础设施（HumanEval+等）
- **UC Berkeley**：Gorilla等工具调用代码评测研究

## 代码 Agent 的轨迹多样性扩展

[[td-scaling]]（2026）提出 TDScaling 框架，核心发现是：对于代码 Agent，训练轨迹的**多样性**比数量更能带来性能增益。通过合成并筛选高差异性的失败/成功轨迹（而非简单扩大数据量），模型性能得到显著提升。这对代码 Agent 评测的启示是：eval 结果需要区分"训练数据量大"和"训练数据多样"这两种改进路径。
