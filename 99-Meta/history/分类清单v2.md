# LLM 评测 Wiki — 分类清单 v2

> v2 与 v1 的差异: v1 是"人类可读的目录树", v2 是 **Karpathy LLM Wiki 三层架构** — 把 wiki 设计成 AI agent 能持续维护的"复利知识库", 同时兼容对外发布。

---

## Context: 为什么对齐 Karpathy 架构

Andrej Karpathy 2026/04 发布的 [LLM Wiki Gist](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f) 提出了一个核心范式: 与其让 LLM 反复 RAG 原始文档, 不如**让 LLM 先把零散信息蒸馏成结构化 wiki, 然后查 wiki**。这个模式天然契合用户需求 (后期写 AI 程序自动维护), 因此 v2 全面对齐。

**核心理念**:
- wiki 是**复利资产**: 每个新源进来, wiki 变得更密、更聪明
- 人类负责: **筛源、提问、审稿**
- LLM 负责: **写、更新、维护交叉引用、发现矛盾**

---

## 核心架构: 三层 + 三特殊文件

```
LLM-Eval-Wiki/
├── CLAUDE.md                  # [Schema 层] 给 AI 的操作手册 (单一真相源)
├── index.md                   # [导航] 全量目录 + 一句话摘要, 每次 ingest 必更
├── log.md                     # [审计] append-only, 记录每次 ingest/query/lint
│
├── raw/                       # [Layer 1] 不可变原始材料 — 人类放, LLM 只读
│   ├── papers/                #   论文 PDF (按 arxiv_id 命名)
│   ├── leaderboards/          #   榜单 JSON 快照 (YYYY-MM-DD/源名.json)
│   ├── articles/              #   博客/news 全文归档
│   ├── reports/               #   AI Index / HELM 等机构报告
│   └── assets/                #   所有图片本地化 (LLM 不能可靠读远程图)
│
├── wiki/                      # [Layer 2] LLM 维护的结构化知识 — LLM owns
│   ├── benchmarks/            #   每个评测集一篇 (wiki 核心数据库)
│   ├── concepts/              #   能力维度、评测方法、统计概念
│   ├── tools/                 #   评测工具链 (harness, OpenCompass...)
│   ├── leaderboards/          #   榜单条目页 (规则 + 历次快照引用)
│   ├── entities/              #   机构、关键研究者、模型厂商
│   ├── sources/               #   每篇 raw/ 源对应一个摘要页
│   ├── synthesis/             #   跨切面综合 (前沿议题 / 批判 / 综述)
│   └── industry/              #   行业垂类 (代码/医疗/法律/金融/教育)
│
└── publish/                   # [Layer 4 / 对外] Quartz 配置 (只指向 wiki/, 不暴露 raw/)
    ├── quartz.config.ts
    └── quartz.layout.ts
```

> 注: `raw/` **不发布到对外站点**, 仅作为 LLM 的 ground truth 来源。Quartz 只构建 `wiki/`。

## CLAUDE.md (Schema 层) — 必须先写好的内容

> 这是给你后续 AI 程序的"宪法"。不写好它, AI 维护就是无序的。

CLAUDE.md 至少覆盖:

- [ ] **目录布局规则**: 哪类内容放哪 (benchmarks/concepts/entities 边界)
- [ ] **frontmatter schema** (见下文)
- [ ] **命名规范**: 文件名用 kebab-case, benchmarks 用官方名 (MMLU.md, SWE-bench.md)
- [ ] **wikilink 规范**: 第一次提及概念必加 `[[]]`, 不要重复链接同一段
- [ ] **引用规范**: 论文 `[作者 年份]` + arXiv ID, 榜单数据必带快照日期
- [ ] **ingest 流程**: 读源 → 写 sources/<slug>.md → 更新相关 wiki 页 → 更新 index → append log
- [ ] **lint 规则**: 死链、孤儿页、矛盾、`last_verified > 90 天`
- [ ] **风格指南**: 中文优先、客观中立、不带厂商褒贬
- [ ] **合规约束**: 引用许可证、利益冲突披露、版权安全词

## Frontmatter Schema (强制, AI 程序的接口)

### benchmarks/
```yaml
---
title: MMLU
type: benchmark
domain: [knowledge, reasoning]
language: en
year: 2020
authors: [Hendrycks et al.]
arxiv_id: 2009.03300
official_url: https://...
license: MIT
size: 15908
format: multiple-choice
status: active            # active | saturated | contaminated | deprecated
last_verified: 2026-05-04
sources: [[[mmlu-paper]], [[mmlu-pro-paper]]]
---
```

### sources/
```yaml
---
title: "Measuring Massive Multitask Language Understanding"
type: source
source_type: paper        # paper | blog | report | leaderboard-snapshot
url: https://arxiv.org/abs/2009.03300
arxiv_id: 2009.03300
authors: [...]
published: 2020-09-07
ingested: 2026-05-04
license: arXiv non-exclusive
discusses: [[[MMLU]]]
---
```

### concepts/, tools/, entities/, leaderboards/, synthesis/
类似结构, 必带 `last_verified` + `sources`。

## 三个核心操作 (你的 AI 程序实现这三个)

### `/ingest <source-path-or-url>`
1. 读源 (本地文件或抓 URL)
2. 在 `wiki/sources/<slug>.md` 写摘要
3. 识别提到的 benchmarks / concepts / entities / leaderboards
4. **同步更新** 10-15 个相关页 (新条目或追加段落)
5. 更新 `index.md`
6. append `log.md`: `## [2026-05-04] ingest | <title> | touched: 12 pages`

### `/query <question>`
1. 读 `index.md` 找相关页
2. 综合答案 + 引用具体页
3. 若答案值得沉淀, 回填到 `wiki/synthesis/`

### `/lint-wiki`
- 死链 (wikilink 指向不存在文件)
- 孤儿页 (无入链, 可能孤立)
- 矛盾 (同一 benchmark 在两页有不一致数字)
- 过期 (`last_verified > 90d`)
- 缺引用 (claim 没绑定 source)
- 许可证缺失

## 关键约束 (Karpathy 强调, 我们必须执行)

1. **图片本地化**: 所有截图、雷达图、榜单截图必须存到 `raw/assets/`, 不依赖远程 URL — LLM 不能可靠读远程图
2. **单源触达 10-15 页**: ingest 一篇论文应该影响 10+ 页, 这是 wiki 网络密度的健康指标; 若一篇源只动 1-2 页, 说明分类太粗
3. **index.md 是单点真相**: 所有"目录页 / MOC" 不要手写, 全部从 index.md 用 Dataview 派生
4. **log.md 是审计闭环**: 用户能 git diff log 看 AI 一周做了什么, 不放心可以回滚
5. **lint 是质量门**: 没跑 lint 不允许发布; lint 失败必须修复或显式 ack

## 验证 (How to verify v2 works)

1. 用 `/ingest` 跑 5 篇核心论文 (MMLU, SWE-bench, GPQA, HELM, LMSYS Arena), 每篇必须触达 10+ 页, 否则分类有问题
2. `/lint-wiki` 跑通, 0 死链 / 0 孤儿
3. Dataview 查询从 frontmatter 派生表能渲染 (按 type/domain/year 分组)
4. `index.md` 从 0 行涨到 100+ 行后, LLM 仍能基于 index 找到正确页 (验证 Karpathy "moderate scale ~hundreds of pages 不需要 RAG")
5. Quartz 只构建 `wiki/`, 访问站点看不到 `raw/` 内容 (隐私 / 版权安全)
