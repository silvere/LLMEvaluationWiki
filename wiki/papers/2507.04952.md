# ArtifactsBench: Bridging the Visual-Interactive Gap in LLM Code Generation Evaluation

> 提出 ArtifactsBench，通过程序化渲染和时序截图捕捉代码生成的视觉交互效果，用多模态 LLM 评估器实现 94.4% 排名一致性，填补视觉可交互代码评估空白。

## 核心主张
- [CLAIM]: 现有代码评测基准以单元测试通过率为核心指标，完全忽略了视觉可交互性（如网页 UI、数据可视化、动画效果），而这些正是用户实际使用中最关心的代码质量维度。
- [CLAIM]: 通过程序化渲染生成物（artifact）并捕捉其动态行为的时序截图，再结合任务清单引导的多模态 LLM 评估，可以实现与 WebDev Arena 人类排名 94.4% 的一致性，以及与人类专家 90% 以上的评估一致性。
- [CLAIM]: 在 ArtifactsBench 的 1,825 个任务测试中，通用大模型频繁超越专用代码模型，挑战了"代码专用模型在代码任务上必然更优"的假设。

## 方法论摘要

ArtifactsBench 的技术核心在于将代码评估从"静态文本比较"升级为"动态视觉交互评估"。评估流水线分三步：第一步，使用无头浏览器（headless browser）程序化渲染 LLM 生成的代码制品（HTML/JavaScript/CSS 网页、可视化图表等）；第二步，捕捉渲染后的时序截图序列，记录交互前后的视觉状态变化；第三步，将截图序列和详细的任务清单（checklist）一并提交给多模态 LLM 评估器，由其逐项核查任务需求是否得到满足。

基准数据集包含 1,825 个多样化任务，覆盖数据可视化、游戏开发、网页 UI 构建、交互式工具等场景，任务难度从基础到复杂分级设置。任务清单由人工专家撰写，确保评估标准的清晰性和可操作性。

研究同步发布了超过 30 个模型（包括 GPT-4o、Claude 3.5 Sonnet、Gemini 1.5 Pro、CodeLlama 等）的系统性评测结果，并与 WebDev Arena 众包人类偏好数据进行了交叉验证。

## 数据与结果

- 基准任务数量：1,825 个
- 测试模型数量：超过 30 个
- 与 WebDev Arena 排名一致性：94.4%
- 与人类专家评估一致性：超过 90%
- 核心发现：通用模型（如 GPT-4o、Claude）频繁超越代码专用模型

## 局限与争议

- 渲染和截图依赖特定的前端技术栈（HTML/JS/CSS），对后端代码、命令行工具、数据处理脚本等无法直接应用此评估方式。
- 多模态 LLM 评估器本身存在偏好偏差，可能对与其训练风格相近的视觉输出打分更高。
- 时序截图捕捉的动态行为有限，复杂的交互逻辑（如多步骤表单填写、游戏状态机）可能无法被完整捕捉。
- 人工撰写的任务清单难以做到完备，评估者 LLM 对清单理解的差异会影响评估结果的稳定性。
