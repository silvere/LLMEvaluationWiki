# SWE-bench: Can Language Models Resolve Real-World GitHub Issues?

> 提出 SWE-bench，包含来自真实 GitHub 仓库的 2,294 个软件工程问题，要求模型生成代码补丁解决真实 issue，评测 LLM 的实际软件工程能力。

## 核心主张
- [CLAIM]: SWE-bench 从 12 个流行 Python 开源仓库（如 Django、Flask、NumPy、scikit-learn 等）收集了 2,294 个 issue-pull request 对，每个任务要求模型生成能通过对应测试套件的代码补丁。
- [CLAIM]: 论文发布时，最强模型（Claude-2）的 % Resolved 仅为 4.8%，GPT-4 为 1.7%，说明真实世界软件工程任务远超当时模型能力。
- [CLAIM]: SWE-bench 的难度来源于：需要跨文件理解大型代码库、定位相关代码位置、正确修改代码而不引入新 bug，比单函数生成复杂得多。
- [CLAIM]: 作者提出"oracle"变体（SWE-bench Oracle），向模型提示正确文件位置，用于分析模型的局限来源是定位还是修复。

## 方法论摘要

数据收集自 GitHub，筛选标准：issue 有对应 PR、PR 包含测试代码、测试在合并前失败而合并后通过。评测时，模型接收问题描述和代码库，生成统一 diff 格式补丁，在隔离环境中运行原始测试套件验证。

## 数据与结果

- 总任务数：2,294 个（后续 SWE-bench Verified 子集 500 个）
- 12 个 Python 仓库
- Claude-2 % Resolved：4.8%（论文发布时）
- GPT-4（with retrieval）% Resolved：1.7%
- Oracle 设置下（已知相关文件）：Claude-2 约 11.3%
- 随后 SWE-agent 等方法将成绩提升至 12-18%+

## 局限与争议

- 测试套件覆盖率有限，补丁"通过测试"不等于完全正确。
- 评测成本高，需要运行大规模 Docker 容器，难以快速迭代。
- 主要覆盖 Python 生态，不代表其他编程语言的能力。
- 模型可能在预训练中见过相关 issue 或 PR，存在数据污染风险。
