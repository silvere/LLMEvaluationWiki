# 多轮对话评测（Multi-Turn Evaluation）

> 评测 LLM 在多轮对话交互中保持上下文理解、信息一致性和任务连贯性能力的评测方法，比单轮评测更接近真实使用场景。

## 定义

多轮对话评测（Multi-Turn Evaluation）针对对话历史包含多个用户-助手交互回合的场景，评测模型在以下维度的能力：

- **上下文维持**：是否记住对话历史中的关键信息
- **指代消解**：正确理解"它"、"上面提到的"等指代
- **任务积累**：在多轮中逐步构建复杂任务的解决方案
- **立场一致性**：不因轮次增加而出现自相矛盾
- **用户意图理解**：跟踪用户目标随对话演进的变化

## 重要性（在 LLM 评测中）

大多数实际 LLM 应用是多轮对话，单轮评测基准（如 MMLU、HumanEval）存在重大盲区：

1. **实际使用差距**：ChatGPT、Claude 等产品的核心使用场景是多轮对话，单轮评测无法预测真实体验
2. **长上下文能力验证**：多轮对话是检验模型上下文窗口利用效率的自然场景
3. **对齐的长期稳定性**：模型可能在对话初期遵守规则，但经过多轮后被逐渐引导违反原则
4. **错误累积效应**：多轮中的错误可能被后续轮次放大，单轮评测无法发现

## 主要方法/实现

**MT-Bench（Zheng et al., 2023）**：
- 80 道题，每题两轮对话，第二轮基于第一轮回答进行追问
- 第二轮题目专门测试上下文依赖（如"根据你上面给的代码，添加错误处理"）
- GPT-4 评分，每轮 1-10 分

**MT-Bench 评测格式**：
```
轮次 1:
User: 为什么天空是蓝色的？
Assistant: [模型回答]

轮次 2:
User: 那为什么日落时天空是橙红色的？  ← 依赖第一轮上下文
Assistant: [模型回答]  ← 如果忘记第一轮，此轮可能不连贯
```

**对话一致性检测**：
- 在多轮中询问相同或等价问题，检测模型回答是否一致
- 在后期轮次中引入与前期矛盾的信息，观察模型如何处理

**User Simulator**：使用 LLM 模拟用户进行真实对话流，评测在自然对话条件下的质量。

## 局限与挑战

- **评测成本高**：多轮评测需要构造完整对话流，比单轮评测昂贵 k 倍
- **第一轮错误的传播**：若第一轮模型回答有误，后续轮次的评测受到污染
- **对话路径爆炸**：真实多轮对话的路径数指数级增长，覆盖性测试不现实
- **人工评测的一致性挑战**：多轮对话的评分需要标注者阅读完整历史，增加认知负担
- **LLM 裁判的上下文理解**：裁判模型也需要理解对话历史才能公平评分，增加评测复杂性
