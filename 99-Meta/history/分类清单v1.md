# LLM 评测 Wiki — 分类清单 v1 (待勾选)

## Context (为什么先做分类)

分类是这个 wiki 的**底层逻辑与顶层设计**。内容会由后续搭建的 AI 程序自动生产与更新，所以分类必须：

1. **MECE** — 互斥且穷尽，不让 AI 把同一个 benchmark 塞到三个目录
2. **结构化** — 每个条目有明确 frontmatter schema，AI 能按字段填充
3. **稳定** — 未来 2 年 LLM 新发展 (Agent/Reasoning/Embodied) 能平滑加入，不需要大重构
4. **客观中立** — 所有条目只呈现事实+引用，不做厂商排名褒贬
5. **独立项目** — 不依赖任何外部 skill，自成闭环

## 分类设计原则

### 每个原子笔记的强制 frontmatter

```yaml
---
title: MMLU
type: benchmark          # benchmark | method | tool | leaderboard | concept | issue
domain:                  # 能力维度标签 (可多)
  - knowledge
  - reasoning
language:                # en | zh | multilingual
year: 2020
authors: [Hendrycks et al.]
arxiv_id: 2009.03300
official_url: https://...
license: MIT
size: 15908              # 题目数
format: multiple-choice  # mc | free-form | code | agent | chat
status: active           # active | deprecated | saturated | contaminated
last_verified: 2026-04-24
---
```

> 这个 schema 是给 AI 程序吃的。frontmatter 稳定，AI 自动更新才能闭环。

## 与 AI 自动更新程序的接口契约

> 后续写的 AI 程序要吃这个 wiki，必须有契约，否则闭环跑不起来。

- **输入**: 爬取 arXiv / HuggingFace / Papers with Code 的新 benchmark，或主流榜单的新一期数据
- **输出**: 按上面 frontmatter schema 生成/更新原子笔记
- **约束**:
  - 每次更新必须附 `source_url` + `retrieved_at` 时间戳
  - 榜单快照以 JSON 存在 `Assets/snapshots/YYYY-MM-DD/`，笔记用 Dataview 查询
  - 引用的论文必须带 arXiv ID + license 字段
  - 厂商数据必须标原始来源，不改写、不添加评价
- **Meta/schema.md** 作为 AI 程序的单一真相源，schema 变更 → 程序更新

## 验证 (How to verify the classification works)

1. 用首期 20 篇核心 benchmark 填到分类里，验证没有"放哪都对 / 放哪都不对"的条目
2. 让 Dataview 按 `type` / `domain` / `year` 聚合，看每个维度是否都能独立成表
3. 图谱视图检查双链网络: 核心节点应是 MMLU / Arena / HELM，成辐射状
4. 请 3 位测试读者按 "决策者 / 研究者 / 学习者" 三条路径找 3 个问题的答案，测试信息架构的可用性
